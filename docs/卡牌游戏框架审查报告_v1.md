## 三、设计缺陷分析

### 3.1 架构层面

#### <u>缺陷1: 类型定义分散</u>
**问题**: 卡牌类型定义在多个文件中重复
- `src/core/types.ts` - CardDefinition
- `src/types/cards.ts` - Card
- `src/battleV2/types.ts` - DebateCard

**影响**: 维护困难，容易产生不一致

**建议**: 
```typescript
// <u>建议统一为单一Card类型，通过扩展实现差异化</u>
interface BaseCard {
  id: string;
  name: string;
  cost: number;
  // ...通用字段
}

interface BattleCard extends BaseCard {
  // 战斗专用字段
}

interface CollectionCard extends BaseCard {
  // 收藏专用字段
}
```

---

#### <u>缺陷2: 状态管理混合</u>
**问题**: 存在两套战斗状态系统
- `src/core/types.ts` - GameState
- `src/battleV2/types.ts` - DebateBattleState

**影响**: 代码冗余，逻辑分散

**建议**: <u>逐步迁移至battleV2系统，废弃旧系统</u>

---

### 3.2 功能层面

#### <u>缺陷3: 效果引擎过于简单</u>
**问题**: `effectEngine.ts` 使用正则解析文本效果，扩展性差

**现有代码**:
```typescript

if (drawMatch) {
  const count = parseInt(drawMatch[1]);
  // ...
}
```

**建议改进**:
```typescript
// <u>建议改为结构化效果定义</u>
interface Effect {
  type: EffectType;
  target: TargetSelector;
  value: number | ValueCalculator;
  condition?: Condition;
  modifiers?: EffectModifier[];
}

// 使用示例

  type: 'draw',
  target: { type: 'self' },
  value: 2,
  condition: { type: 'has_tag', tag: '策略' }
};
```
