> 适用范围：本仓库所有 AI（Codex / Claude / Gemini / 其他执行模型）

## 三、仲裁决策矩阵（快速参考）

| 冲突类型 | 优先级排序 | 关键考量点 | 决策倾向 |
|----------|------------|------------|----------|
| 接口设计 | 正确性 > 扩展性 > 性能 | 可测试性、向后兼容 | 优先选择接口更清晰的方案 |
| 实现方式 | 可维护性 > 性能 > 快速实现 | 代码复杂度、调试难度 | 优先选择更易理解的方案 |
| 数据结构 | 内存占用 > 序列化性能 > 查询效率 | Unity序列化限制、Editor体验 | 优先选择序列化友好的方案 |
| 性能优化 | 可测量性 > 优化幅度 | 是否有Profiler数据 | 优先选择有数据支撑的方案 |
| 扩展性 | 解耦度 > 灵活性 > 复杂度 | 接口数量、依赖关系 | 优先选择依赖注入/事件系统 |
| 命名规范 | 一致性 > 语义化 | 是否符合C#规范 | 优先选择团队统一规范 |

## 四、预设冲突场景决策树

```text

    │
    ├─ 是否影响核心功能正确性？
    │   ├─ 是 → 选择"更稳定、更可测试"的方案
    │   └─ 否 ↓
    │
    ├─ 是否涉及性能优化？
    │   ├─ 是 → 是否有Profiler数据支撑？
    │   │   ├─ 有 → 选择数据更优的方案
    │   │   └─ 无 → 先做性能测试再决策
    │   └─ 否 ↓
    │
    ├─ 是否涉及扩展性？
    │   ├─ 是 → 选择"接口更清晰、耦合更松"的方案
    │   └─ 否 ↓
    │
    ├─ 是否涉及实现复杂度？
    │   ├─ 是 → 选择"更易理解、更易维护"的方案
    │   └─ 否 ↓
    │
    └─ 势均力敌 → 选择"更符合团队技术栈"的方案
```

## 五、执行补充（本仓库强制）

1. 任何冲突都要先创建 `CONFLICT-[编号]` 记录，再动手改文件。  
2. 需求不清楚时，先问用户，不得自行脑补。  
3. 修正输出必须包含：
   - 修改前 vs 修改后
   - 文件/函数/类名
   - 影响说明
