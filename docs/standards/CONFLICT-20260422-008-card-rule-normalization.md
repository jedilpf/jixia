## 仲裁依据

### 项目优先级排序（按重要性）
1. **核心目标**: 战斗规则可执行、可验证、可落后端。  
2. **用户当轮指令**: 保留两类卡牌并建立有意义的数值对抗。  
3. **风险控制**: 不手工重写全量卡池，采用数据边界归一化。  
4. **可维护性**: 风味文本与规则文本分层，避免后续继续混杂。  

### 决策规则
- 涉及核心功能正确性时，优先“结构化、可测试、可回归”的方案。  
- 涉及历史数据量大时，优先“可批处理归一化”的方案。  

---

## 仲裁决策

### 最终方案
**采用**: 混合方案（保留风味文本 + 新增结构化规则 + 归一化规则文本）

### 决策理由
- 兼顾叙事表现与数值执行：UI 仍可展示风味内容，后端只依赖结构化规则。  
- 通过归一化层统一处理中文大写数字与冗余描述，避免手工改写全量卡牌。  
- 保持两类卡牌（立论/策术）对外口径一致。  

### 实施要求
- [x] `showcaseCards` 增加 `rule` 与 `ruleText` 字段，并保留 `background`。  
- [x] `data/cards.ts` 适配层优先使用 `ruleText` 作为运行时描述输入。  
- [x] battleV2 规则执行支持结构化规则，兼容旧 `effectKind/effectValue`。  
- [x] 卡牌详情与图鉴优先展示规则文本。
