## 📖 为什么命名规范很重要？

### 想象一下这个场景...

**没有规范时**：
```
文件夹里：
├── 战斗设计.md
├── battle_design.md
├── BattleDesign.md
├── 战斗系统设计文档v1.md
└── 战斗-设计-v1.md

问题：哪个是最新的？内容有什么区别？
```

**有规范后**：
```
文件夹里：
├── battle-design-核心规则-v1.2.md  ← 一眼就知道是什么
└── battle-design-机制分析-v1.md    ← 清晰明了
```

### 命名规范的好处

| 好处 | 说明 |
|------|------|
| 🔍 **易于查找** | 看文件名就知道内容 |
| 📋 **避免混乱** | 不会出现多个版本 |
| 🤝 **团队协作** | 大家都用同样的方式命名 |
| 🤖 **AI友好** | AI助手能更好地理解项目 |

## 📝 你需要知道的命名规则

### 文档命名公式

```
模块-类型-主题-版本.md
│     │    │    │
│     │    │    └─ 版本号（可选）
│     │    └─ 中文描述内容
│     └─ 文档类型
└─ 功能模块
```

### 模块名称速查表

| 模块代码 | 含义 | 例子 |
|----------|------|------|
| `battle` | 战斗相关 | 战斗规则、战斗机制 |
| `card` | 卡牌相关 | 卡牌设计、卡牌平衡 |
| `faction` | 门派相关 | 门派设定、阵营设计 |
| `ui` | 界面相关 | 界面设计、交互设计 |
| `project` | 项目管理 | 开发计划、需求文档 |
| `dev` | 技术开发 | 技术架构、开发指南 |

### 文档类型速查表

| 类型代码 | 含义 | 什么时候用 |
|----------|------|-----------|
| `design` | 设计文档 | 描述设计方案、游戏机制 |
| `spec` | 规格文档 | 描述技术规格、功能规格 |
| `guide` | 指南手册 | 教程、操作指南 |
| `report` | 报告文档 | 测试报告、分析报告 |
| `plan` | 计划文档 | 开发计划、实施计划 |
| `req` | 需求文档 | 需求分析、功能需求 |

## 🔍 常见问题解答

### Q1: 版本号怎么写？

**格式**：`v` + 数字

```
v1    第一版
v1.1  小更新
v1.2  小更新
v2    大版本更新
```

### Q2: 文档内容更新了，版本号要改吗？

**是的！** 每次重要更新都要改版本号：

| 更新程度 | 版本变化 | 示例 |
|----------|----------|------|
| 小修改、纠错 | 小数点+1 | v1.0 → v1.1 |
| 新增章节 | 小数点+1 | v1.1 → v1.2 |
| 大改动、重构 | 整数+1 | v1.2 → v2.0 |

### Q3: 文档名太长怎么办？

**保持简洁**，主题部分控制在10个汉字以内：

```
✅ battle-design-核心规则-v1.md
❌ battle-design-这是关于战斗系统核心规则的详细说明文档-v1.md
```

### Q4: 旧文档怎么处理？

1. **不要删除** - 移动到 `docs/archive/`
2. **保留原文件名** - 方便追溯
3. **创建新版本** - 按规范命名
