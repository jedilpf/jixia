## 仲裁依据

### 项目优先级排序
1. **核心目标**: 保持 `battleV2` 的原创战斗骨架与三阶卡牌主链路一致，不引入高相似结构。
2. **技术约束**: 当前仓库已经存在 `starTier/unlockLevel/deck quota` 体系，且本周不宜新增独立机制轨。
3. **时间限制**: 当前优先事项仍是可玩性与数值收敛，不是重做成长模式。
4. **扩展性要求**: 未来若增强三阶体验，应复用现有 `playerLevel/topic/writing/phase` 体系，而非平行再造一个外部式升级循环。
5. **团队能力**: 当前代码和文档已经围绕现有战斗阶段与等级系统展开，延续成本最低。

### 决策规则
- 本次冲突同时涉及核心结构正确性与版权风险，优先选择“更稳定、更可测试、与现有系统更一致”的方案。
- 外部参考只允许抽象到“节奏原则”和“信息架构层级”，不允许照抄具体循环、命名、资源结构或升级触发方式。

## 仲裁决策

### 最终方案
**采用**: 混合方案

### 决策理由
- 保留仓库既有三阶主方案: `1星/2星/3星` 继续由账号等级门槛和构筑配额驱动。
- 允许吸收《月圆之夜》的三个抽象原则:
  1. 高阶内容要“逐步开放”，不能在早期一次性全部进入。
  2. 强卡运作要有清晰、少量、可预判的触发窗口。
  3. 成长收益要和风险/节奏绑定，而不是纯数值堆叠。
- 明确拒绝直接复用以下结构:
  - 商店刷新/商店升级/经验买等级
  - 三连合成/金卡升级/joker 替代
  - 六格自走棋站位与对应攻击序列
  - 经典模式的行动力/法力双资源卡型分层
- 若后续要强化“三阶卡如何在战斗里运作”，只能挂接到我们已有结构:
  - `明辩 -> 暗谋 -> reveal -> resolve`
  - `topic`
  - `writing`
  - `playerLevel`
  - `deck tier quota`
