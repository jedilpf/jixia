## 二、文档分类索引系统

### 2.1 文档分类标签

每份文档应在开头包含元数据标签：

```markdown
---
document_id: DOC-BATTLE-001
category: design
module: battle
status: active
version: 1.2
created_at: 2026-03-15
updated_at: 2026-03-23
author: design-team
tags: [战斗, 机制, 核心规则]
related_docs: [DOC-CARD-001, DOC-UI-002]
---
```

### 2.2 文档ID分配规则

```
DOC-{模块}-{序号}

模块标识:
- BATTLE    战斗系统
- CARD      卡牌系统
- FACTION   门派系统
- UI        用户界面
- PROJECT   项目管理
- DEV       开发技术
- SYSTEM    通用系统
- META      元文档
```

### 2.3 文档状态定义

| 状态 | 说明 | 可见性 |
|------|------|--------|
| `draft` | 草稿，内容未完成 | 内部 |
| `review` | 待审核，内容已完成待确认 | 内部 |
| `active` | 活跃，当前有效文档 | 公开 |
| `deprecated` | 废弃，已被新版本替代 | 归档 |
| `archived` | 归档，历史参考 | 归档 |

## 三、版本控制协议

### 3.1 版本号规范

```
v主版本号.次版本号.修订号

示例: v1.2.3
- 主版本号(1): 重大变更，不兼容更新
- 次版本号(2): 功能增加，向后兼容
- 修订号(3): 问题修复，向后兼容
```

### 3.2 版本更新规则

| 变更类型 | 版本更新 | 示例 |
|----------|----------|------|
| 新建文档 | v1.0.0 | 首次创建 |
| 新增章节 | v1.x.0 | v1.0.0 → v1.1.0 |
| 内容修正 | v1.x.x | v1.1.0 → v1.1.1 |
| 结构重组 | v(x+1).0.0 | v1.2.3 → v2.0.0 |
| 废弃归档 | 保留原版本号 | 移至archive/ |

### 3.3 变更记录格式

每份文档应在末尾包含变更记录：

```markdown
## 变更记录

| 版本 | 日期 | 变更内容 | 作者 |
|------|------|----------|------|
| v1.2.0 | 2026-03-23 | 新增第三章：命名示例 | 张三 |
| v1.1.0 | 2026-03-20 | 补充目录结构说明 | 李四 |
| v1.0.0 | 2026-03-15 | 初始版本 | 王五 |
```

## 五、文档生命周期管理

### 5.1 生命周期阶段

```
创建 → 审核 → 发布 → 维护 → 废弃 → 归档
```

### 5.2 阶段转换规则

| 当前阶段 | 下一阶段 | 触发条件 |
|----------|----------|----------|
| 创建 | 审核 | 内容完成 |
| 审核 | 发布 | 审核通过 |
| 发布 | 维护 | 内容更新 |
| 维护 | 废弃 | 新版本替代 |
| 废弃 | 归档 | 超过30天 |

### 5.3 文档归档规则

归档条件：
- 被新版本完全替代
- 超过90天未更新且非活跃文档
- 项目决策变更导致文档失效

归档操作：
1. 保留原文件名和版本号
2. 移动至 `docs/archive/old-versions/` 或 `docs/archive/deprecated/`
3. 在原位置创建重定向说明
4. 更新主索引
