# 《百家争鸣》“明谋暗辩”机制开发说明（给 Codex / Anti Gravity）

## 1. 用一句话定义这个机制

**明谋暗辩 = 双方同时决定“把哪张牌放到哪个槽位”，但在揭示前，只能看到对方“放了牌、放在哪个槽位”，看不到具体是什么牌。**

也就是说：

- **明谋**：你公开你的意图方向  
  - 你要抢主议
  - 你要经营旁议
  - 你这回合不出牌
- **暗辩**：你不公开具体手牌内容  
  - 对方知道你把牌放去了主议
  - 但不知道你放的是哪张牌

---

## 2. 这个机制到底想解决什么问题

它要解决的不是“轮流出牌太慢”，而是让游戏更像：

- 预判
- 设局
- 猜位
- 心理战

所以这个模式下，玩家真正做的不是“看见对方出了什么再反应”，而是：

**根据议题、门派、当前场面，猜对方这一手会在哪一路发力。**

---

## 3. 先定死：这一版不是轮流出牌

这套机制下，**不是 A 先出牌、B 后出牌**。  
而是：

### 每个回合分 4 个大阶段
1. **双方同时提交出牌**
2. **双方同时揭示**
3. **按固定顺序结算**
4. **进入回合结果与下一回合**

所以它本质上是：

**同步提交 + 顺序结算**

不是“完全同时乱算”，也不是“传统轮流行动”。

---

## 4. 每回合玩家到底能做什么

## 4.1 每回合，每个玩家只能做 1 个“行动声明”
首版先这样定，最容易实现：

每回合玩家只能从下面 4 种里选 1 种：

1. **向主议放 1 张牌**
2. **向旁议放 1 张牌**
3. **向预备区放 1 张施策**
4. **空过**

> 注意：首版明谋暗辩模式，先不要做“每回合 2 张牌”。  
> 先做成“每回合 1 个同步声明”，这样规则最清楚，程序最稳。

---

## 4.2 玩家在客户端上看见什么

在“提交阶段”，玩家界面上会看到：

- 自己手牌
- 自己当前主议立论 / 旁议立论
- 对方当前主议立论 / 旁议立论（已经在场上的公开信息）
- 本回合 3 个可提交槽位：
  - 主议提交区
  - 旁议提交区
  - 预备区提交区
- 一个“空过”按钮
- 一个“确认提交”按钮

### 玩家提交后，对方只能看见：
- 你是否已经提交
- 你提交到了哪个区：
  - 主议
  - 旁议
  - 预备区
  - 空过

### 对方不能看见：
- 你具体提交了哪张牌
- 你这张牌的数值
- 你这张牌的效果

---

## 5. 服务端一回合的标准结算流程

下面这个顺序要写死。  
**整个机制能不能稳定，全靠这里的顺序清不清楚。**

---

## Phase 0：回合开始
服务器进入新回合时，做这些事：

1. 回合数 +1
2. 双方同时抽 1 张牌
3. 更新本回合费用上限
4. 重置“本回合临时状态”
5. 广播给双方：
   - 当前回合数
   - 当前费用
   - 当前手牌数
   - 当前场上牌面

### 费用曲线
沿用你现在的想法：

- 第 1 回合：2
- 第 2 回合：3
- 第 3 回合：4
- 第 4 回合及以后：5

---

## Phase 1：同步提交
服务器给双方开启一个倒计时，比如 20 秒。

双方在这 20 秒内，同时提交一个 Action：

```ts
type PlannedAction =
  | { type: "play_card", cardId: string, zone: "main" | "side" | "prep" }
  | { type: "pass" }
```

### 合法性检查
服务器在收包时先只做基础校验：

- 这张牌是否还在该玩家手里
- zone 是否合法
- 该牌类型和 zone 是否匹配
  - 立论只能进 main 或 side
  - 施策可以进 main / side / prep（首版这样定）
- 当前 zone 是否允许放置
- 当前用度是否足够

### 超时处理
如果玩家超时未提交：

- 默认视为 `pass`

---

## Phase 2：锁定提交结果
20 秒结束后，服务器锁定双方提交内容。

然后向双方广播一条**简报信息**：

```ts
{
  playerA: { submitted: true, zone: "main" },
  playerB: { submitted: true, zone: "side" }
}
```

### 注意
这里**仍然不公开 cardId**。  
只公开“对方这回合把牌放去了哪里”。

这一步就是“明谋”。

---

## Phase 3：同时揭示
锁定后，服务器同时公开双方提交的真实内容。

广播示例：

```ts
{
  reveal: [
    { player: "A", cardId: "card_102", zone: "main" },
    { player: "B", cardId: "card_587", zone: "side" }
  ]
}
```

从这一刻开始，这两张牌才变成公开信息。

这一步就是“暗辩揭示”。

---

## Phase 4：按固定顺序结算
这一步不能模糊，必须写死。

### 推荐顺序
1. **立论进场**
2. **施策生效**
3. **议位胜负结算**
4. **根基变化 / 离场**
5. **大势 / 筹更新**
6. **胜利检查**

---

## 6. 立论如何进场

### 6.1 进场规则
如果本回合揭示的是立论牌：

- zone = main → 进入主议槽位
- zone = side → 进入旁议槽位

### 6.2 冲突规则
每个玩家每个议位最多只能有 1 张立论。

如果该玩家该议位已经有立论，则首版建议这样处理：

**新牌不能覆盖旧牌，提交非法，改为本回合空过。**

也就是说，首版不要做“替换进场”，不然复杂度会上去。

---

## 7. 施策如何结算

施策是这套机制最容易让开发看不懂的地方，所以我直接给你一个最简单可做的版本。

## 7.1 施策必须在提交时就声明目标议位
也就是说，施策不能写成：

- 选择一张敌方立论……

首版要改成：

- 选择主议或旁议，该议位目标……

### 例子
旧写法：
- 选择一张敌方立论，本回合 -2 辩锋

明谋暗辩版改写：
- **若你将此牌提交至主议：敌方主议立论本回合 -2 辩锋**
- **若你将此牌提交至旁议：敌方旁议立论本回合 -2 辩锋**

这样程序才能在揭示后直接结算，不需要临场再弹额外选择框。

---

## 7.2 施策的 zone 语义

### zone = main
表示该施策作用于主议

### zone = side
表示该施策作用于旁议

### zone = prep
表示该施策是资源类 / 全局类 / 抽牌类 / 不绑定具体议位的牌

---

## 7.3 施策和立论同时揭示时，先后怎么判
首版统一定成：

- **先让立论进场**
- 再让施策结算

这样施策才有目标，不容易空放。

---

## 8. 主议 / 旁议怎么结算

当立论进场、施策都结算完之后，再结算议位。

---

## 8.1 主议结算
比较双方主议立论的最终辩锋。

- 高者赢主议
- 赢者 +1 大势
- 输者主议立论 -1 根基

---

## 8.2 旁议结算
比较双方旁议立论的最终辩锋。

- 高者赢旁议
- 赢者获得 1 个筹
- 每人最多持有 1 个筹
- 输者旁议立论 -1 根基

---

## 8.3 平局
如果某一议位双方辩锋相等：

- 该议位无人获胜
- 双方该议位立论各 -1 根基

---

## 9. 筹怎么用

你之前的想法是“旁议赢了给一个资源”。  
在明谋暗辩里，这个资源更应该简单，不要复杂。

### 首版建议
- 赢旁议 → 获得 1 个筹
- 持有上限 1
- 在 **Phase 1 同步提交前**，玩家可以声明是否消耗 1 个筹
- 若消耗，则本回合提交的那张牌 -1 用度

也就是说，客户端在提交动作里应变成：

```ts
type PlannedAction =
  | { type: "play_card", cardId: string, zone: "main" | "side" | "prep", useToken: boolean }
  | { type: "pass" }
```

---

## 10. 胜利条件
建议首版先保守一点：

- **先到 8 大势获胜**

因为明谋暗辩的回合更紧凑、博弈更硬，  
8 大势已经够打出节奏了。

---

## 11. 这个机制下，哪些卡能直接用，哪些不能直接用

这是给开发和策划一起看的。

---

## 11.1 可以直接用的牌
这类牌最安全：

### A. 纯立论牌
例如：
- 1/2 白板
- 若置于主议，本回合 +1 辩锋
- 进场时 +1 根基

### B. 纯资源牌
例如：
- 抽 2 弃 1
- 下一张牌 -1 用度
- 获得 1 个筹

### C. 绑定议位的简单施策
例如：
- 提交到主议：敌方主议 -1 辩锋
- 提交到旁议：己方旁议 +1 根基

---

## 11.2 需要改写的牌
原来写“选择一张敌方立论”的，基本都要改成“绑定主议/旁议”。

### 改写原则
旧：
- 选择一张敌方立论：文本失效

新：
- 若此牌提交到主议：敌方主议立论文本失效
- 若此牌提交到旁议：敌方旁议立论文本失效

---

## 11.3 首版建议禁用的牌
这些牌先不要进明谋暗辩实验池：

- 结算时再选择具体对象的反制牌
- “若本回合该牌已经获得过增益”这类需要回看日志的牌
- 复杂移动牌
- 同时影响两个议位并且依赖先后顺序的牌
- 需要玩家在揭示后再做二次目标选择的牌

---

## 12. 客户端界面怎么做

## 12.1 战斗主界面
### 中间战场
- 上方：对方主议 / 旁议
- 下方：己方主议 / 旁议

### 底部
- 手牌区（5 张起手，后续摸牌补充）
- 当前费用
- 当前筹数量
- 提交按钮
- 空过按钮

### 中部右侧
- 倒计时
- 本回合状态说明：
  - 等待提交
  - 已提交，等待对手
  - 揭示中
  - 结算中

---

## 12.2 提交交互
玩家操作流程：

1. 点击一张手牌
2. 点击“主议 / 旁议 / 预备区”之一
3. 牌飞入对应提交槽位（此时对自己可见牌名，对对手不可见）
4. 可选择是否使用 1 个筹减费
5. 点击“确认提交”
6. 确认后锁定，不能更改

---

## 12.3 对手视角
对手只看到：

- 你在主议槽位盖了一张牌
- 或你在旁议槽位盖了一张牌
- 或你在预备区盖了一张牌
- 或你空过

看不到具体是什么。

---

## 13. 最小可实现版本（MVP）
如果 Codex 现在还是觉得复杂，就先按这个砍到最小：

### MVP 规则
- 每回合双方各提交 **1 张牌**
- 只允许放：
  - 主议
  - 旁议
  - 空过
- **不开放预备区**
- **不开放复杂施策**
- 只开放三类牌：
  1. 立论牌
  2. 简单增减辩锋牌
  3. 简单抽弃牌

### 为什么这样做
因为你要先验证：

**“同时声明槽位、隐藏具体牌名”这件事到底好不好玩。**

不是一上来就验证全卡池。

---

## 14. 给 Codex 的最终实现理解

如果要把这套机制翻译成一句开发语言，就是：

> 这是一个 **双人同步提交动作、延迟公开卡面、固定顺序结算结果** 的卡牌对战系统。

他们只要先实现下面这 5 件事，机制就跑起来了：

1. 双方同时提交一个 Action
2. 提交阶段只广播 zone，不广播具体 cardId
3. 时间到后统一 reveal
4. 统一按固定顺序结算
5. 回合结束后更新大势 / 根基 / 筹

---

## 15. 最后的拍板建议

不要让 Codex 一上来做“完整明谋暗辩正式版”。  
先让他们做：

### `battle_mode = hidden_slot_test`
也就是一个实验战斗模式：

- 同步提交
- 隐藏牌名
- 公开槽位
- 揭示结算
- 小牌池

先把这个模式跑通。  
跑通以后，再决定要不要把整套《百家争鸣》主规则切到这上面。
