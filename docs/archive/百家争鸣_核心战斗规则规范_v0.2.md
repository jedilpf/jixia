# 《百家争鸣》核心战斗规则规范 v0.2
## ——给程序实现的清晰版

## 1. 文档目的
这份文档不是世界观文案，而是**程序可实现的战斗规则规范**。  
目标是让代码编写工具或开发者能够直接按规则拆状态、拆流程、拆数据结构。

---

# 2. 核心定义

## 2.1 游戏核心
《百家争鸣》的战斗不是普通卡牌对打，而是：

**围绕一个中央公共议题，进行“暗辩提交 → 明辩揭示 → 议题推进 → 周期性引爆”的叙事辩论式卡牌战斗。**

## 2.2 三层结构
游戏的战斗由三层构成：

### 第一层：表层战斗
- 出牌
- 比拼主议 / 旁议
- 获得大势
- 场上单位离场或存活

### 第二层：明辩
- 双方同步翻开本轮打出的牌
- 玩家能够看见对方本轮具体打了什么牌
- 这些牌代表对当前议题的公开立场和观点

### 第三层：暗辩
- 在揭示前，玩家只能看到对方“出了没有、出了几张、放在哪个议区”
- 看不到对方具体牌名
- 暗辩的本质不是隐藏数值，而是**隐藏本轮真实选择与组合意图**

## 2.3 议题
议题不是开局固定 Buff，也不是局前配置项。  
议题是战斗中间的**公共争夺核心**。

一局战斗只有**1 个中央议题**。  
这个议题会经历三个阶段：

1. **议题种子**：刚开始，只是一个模糊议题
2. **议题争夺**：双方出牌不断推动议题方向
3. **议题引爆**：每隔固定轮次判定一次，若条件达成则触发高光演出和强战斗效果

# 3. 本局基础规则

## 3.1 对战人数
- 1v1

## 3.2 起手手牌
- 每位玩家起手 5 张牌

## 3.3 牌组构成
当前先用固定预组牌组，不开放自定义构筑。

推荐默认：
- 门派牌 12 张
- 通用牌 12 张
- 合计 24 张

## 3.4 费用曲线
每轮最大用度如下：

- 第 1 轮：2
- 第 2 轮：3
- 第 3 轮：4
- 第 4 轮及以后：5

## 3.5 胜利条件
- 先到 8 大势者获胜

---

# 4. 场面结构

## 4.1 战斗场地只有两个议区
每位玩家都有两个战斗区：

- **主议**
- **旁议**

## 4.2 不使用左中右三个席位
当前版本明确规定：

- **不使用左 / 中 / 右 三个席位**
- 只保留两个议区：主议、旁议
- 每个议区本轮最多放 3 张牌

也就是说：

- 主议区：0 ~ 3 张
- 旁议区：0 ~ 3 张

## 4.3 为什么不做左中右席位
原因：
1. 对当前玩法没有必要
2. 会让 UI 更乱
3. 会让程序状态和判定更复杂
4. 当前真正有意义的是“主议 vs 旁议”，不是更多物理站位

---

# 5. 每轮出牌规则

## 5.1 一轮不是只出 1 张牌
当前版本明确规定：

**每位玩家每轮不是只出 1 张牌，而是可以打出一个“牌组组合”。**

这是为了保证：
- 有组合感
- 有操作爽感
- 能体现门派套路
- 能做类似“连招 / 牌组配合”的感觉

## 5.2 每轮出牌上限
当前建议先定为：

- 每位玩家每轮最多打出 **3 张牌**
- 这 3 张牌可以自由分配到：
  - 主议
  - 旁议

例如：
- 主议 2 张，旁议 1 张
- 主议 3 张，旁议 0 张
- 主议 1 张，旁议 2 张
- 主议 0 张，旁议 3 张
- 或本轮不出牌

## 5.3 费用限制
玩家本轮打出的全部牌，其总费用必须 **<= 本轮可用用度**。

## 5.4 施策牌限制
当前版本先不单独限制“施策最多几张”。  
是否要限制，后续再平衡。

程序上当前可先做成：
- 只要费用够、手里有牌、总张数不超过 3，就可提交

# 6. 明辩 / 暗辩流程

## 6.1 一轮流程总览
每轮固定分为 7 个阶段：

1. 回合开始
2. 暗辩提交
3. 提交锁定
4. 明辩揭示
5. 公开效果结算
6. 议区胜负结算
7. 议题推进 / 议题引爆判定

## 6.2 阶段 1：回合开始
系统执行：
- 回合数 +1
- 当前玩家抽牌
- 恢复本轮最大用度
- 清理“本轮结束后失效”的状态
- 刷新 UI 提示

> 当前继续使用“轮流完整回合制”。

## 6.3 阶段 2：暗辩提交
玩家进入本轮暗辩阶段。

### 规则
- 玩家从手牌中选择 0 ~ 3 张牌
- 决定每张牌放入：
  - 主议
  - 旁议
- 所有选择都以**暗置**方式提交
- 在双方都确认前，看不到对方具体牌名

### 玩家此时能看见什么
只能看见：
- 对方主议区放了几张背面牌
- 对方旁议区放了几张背面牌
- 对方是否已确认提交

看不见：
- 对方具体打了哪几张牌
- 对方牌的顺序内容
- 对方组合效果

这一步就是“暗辩”。

## 6.4 阶段 3：提交锁定
双方提交完成后，系统锁定本轮出牌。

### 一旦锁定：
- 不能改牌
- 不能改区
- 不能追加
- 不能撤回

## 6.5 阶段 4：明辩揭示
双方同步翻开本轮提交的所有牌。

此时玩家能看见：
- 对方本轮具体打出的牌名
- 这些牌放在主议还是旁议
- 这些牌的公开说明与意象动画

### 明辩的本质
明辩不是单纯翻牌。  
明辩表示：
- 双方对同一个中央议题，做出了各自的公开表态
- 这是一次可见的“辩”

# 7. 牌组组合规则

## 7.1 组合感的定义
本游戏中的“组合”，不是斗地主/麻将那种完全固定牌型规则。  
它的组合感来自于：

1. 同轮可以下多张牌
2. 同议区的多张牌会形成联动
3. 某些牌会读取同轮已打出的其他牌
4. 某些牌会因为同门派 / 同标签 / 同议题方向而获得加成

## 7.2 同议区牌的顺序
主议和旁议内部，不再做“左中右席位”。  
但每个议区内的牌，仍然有一个**提交顺序**。

例如：
- 主议第 1 张
- 主议第 2 张
- 主议第 3 张

程序里可记录为数组顺序：

```ts
mainQueue: CardInstance[]
sideQueue: CardInstance[]
```

这个顺序用于：
- 触发连锁效果
- 解决“先后读取”关系
- 播放动画顺序

但 UI 上不需要刻意做成“左中右站位”。

## 7.3 建议的组合判定类型
为了让代码好写，先只做下面 3 类组合：

### A. 同门派组合
若同一议区内，本轮打出 2 张及以上同门派牌，则触发门派组合加成。

### B. 同议题标签组合
若同一议区内，多张牌的 `issueTag` 相同，则对该议题方向额外推进。

### C. 连锁组合
若某张牌写有：
- “若本轮你已在本议区打出过一张 X 标签牌，则……”
则按顺序触发

> 这 3 类组合足够支持“牌组感”，又不至于太难实现。

# 8. 主议 / 旁议结算规则

## 8.1 每个议区都单独计算“论势值”
主议和旁议各自计算一个总值，暂命名为：

- `mainPower`
- `sidePower`

### 计算方式
该议区本轮所有已生效牌的公开战斗值、效果修正值、组合修正值，累加得到总值。

公式可先写成：

```ts
zonePower = sum(card.publicPower + modifiers + comboBonus)
```

## 8.2 主议结算
比较双方 `mainPower`：

- 高者赢主议
- 赢主议者 +1 大势
- 输主议者，该主议区全部存活牌各 -1 根基

## 8.3 旁议结算
比较双方 `sidePower`：

- 高者赢旁议
- 旁议本身不直接给大势
- 旁议主要用于：
  1. 推进中央议题
  2. 获得暗势优势
  3. 为下轮或引爆做铺垫

# 9. 议题系统

## 9.1 一局只有 1 个中央议题
本局开局时，系统生成 1 个初始中央议题。  
这个议题一开始只是“种子态”，尚未完全定型。

例如：
- 治国先富民还是先立法
- 危局之下先守还是先变
- 民意应安抚还是应动员

## 9.2 议题不是固定 Buff
议题不是一开始就给固定加成。  
它必须通过战斗过程逐渐被推动。

## 9.3 议题有方向分数
每个中央议题内部，至少记录多个方向分数，例如：

```ts
directionScore = {
  ritual: 0,
  economy: 0,
  strategy: 0
}
```

玩家打出的牌，会推动其中某个或某几个方向。

## 9.4 议题推进来源
当前版本先定为 3 个来源：

### 来源 A：牌本身的议题标签
每张牌有 `issueTag`。  
例如：
- ritual
- economy
- strategy
- neutral

打出这张牌后，对应方向推进 +1 或按牌面值推进。

### 来源 B：明辩胜利
若本轮某方赢得主议，则该方主议区内占主导的议题方向再额外推进。

### 来源 C：旁议胜利
若某方赢得旁议，则该方旁议区内占主导的议题方向再额外推进。

> 这样议题推进就和你的战斗结果直接绑定起来。

# 10. 议题引爆机制

## 10.1 引爆不是每轮都有
当前建议：

**每 3 轮检查一次是否触发议题引爆。**

也就是：
- 第 3 轮检查一次
- 第 6 轮检查一次
- 第 9 轮检查一次

## 10.2 引爆判定方式
引爆不是纯随机，而是：

**蓄力为主，概率为辅。**

系统记录一个 `burstScore`，来源包括：

1. 当前议题热度
2. 当前领先方向与第二方向的差值
3. 当前门派共鸣
4. 当前民众支持
5. 最近三轮议题推进总量

程序上可先写成：

```ts
burstScore =
  heat * 0.4 +
  directionLead * 0.25 +
  support * 0.2 +
  factionResonance * 0.15
```

## 10.3 引爆触发规则
- 若 `burstScore >= 100`：直接引爆
- 若 `70 <= burstScore < 100`：按概率引爆
- 若 `< 70`：本轮不引爆

## 10.4 引爆效果
议题引爆后，触发：

1. 一段大动画
2. 当前议题暂时锁定到领先方向
3. 产生一次强战斗效果
4. 提供阶段性剧情导向
5. 对应门派可获得高光表现

### 强战斗效果的例子
- 当前领先方向的所有牌本轮额外 +X 论势
- 对方全部议区受到压制
- 本轮大势奖励翻倍
- 获得额外民众支持

> 具体数值和效果先用占位，不要在现在死磕平衡。

# 11. 民众支持系统

## 11.1 民众支持的作用
民众支持不是独立小游戏。  
它是“议题是否站在你这边”的外显资源。

玩家通过以下方式获得支持：
- 连续赢得明辩
- 在议题方向上形成明确领先
- 成功触发议题引爆
- 打出高影响力门派组合

## 11.2 民众支持的用途
支持值可以参与：
- 议题引爆判定
- 高光动画触发概率
- 某些门派技能的解锁条件
- 战斗后期的大势加成

# 12. 数据结构建议

## 12.1 卡牌结构
```ts
type Card = {
  id: string
  name: string
  faction: string
  type: "argument" | "scheme" | "support"
  cost: number

  publicPower: number          // 明辩公开战斗值
  issueTags: string[]          // 这张牌倾向推动哪些议题方向
  comboTags: string[]          // 用于组合判定
  effects: CardEffect[]        // 公开效果

  artKey: string               // 动画 / 意象资源
}
```

## 12.2 议题结构
```ts
type IssueState = {
  id: string
  name: string
  stage: "seed" | "contested" | "locked"
  heat: number
  directionScore: Record<string, number>
  triggerThreshold: number
  burstCheckEvery: number // 固定为 3
}
```

## 12.3 战斗状态结构
```ts
type BattleState = {
  round: number
  players: [PlayerState, PlayerState]
  currentIssue: IssueState
  supportScore: Record<string, number>
  factionResonance: Record<string, number>
}
```

## 12.4 玩家战斗区结构
```ts
type DebateBoard = {
  mainQueue: CardInstance[]   // 主议区，本轮最多 3 张
  sideQueue: CardInstance[]   // 旁议区，本轮最多 3 张
}
```

# 13. 当前明确确定的点

以下内容当前已经明确，不要再模糊：

1. **一轮不是只出 1 张牌，而是可打 1 ~ 3 张牌组成组合**
2. **主议和旁议必须保留**
3. **不使用左中右三个席位**
4. **暗辩的本质是隐藏信息，不是隐藏数值**
5. **每 3 轮进行一次议题引爆判定**
6. **议题是战斗中心，不是边角 Buff**
7. **追议题和不追议题都必须成立**

# 15. 给程序实现的最后一句话

当前版本要实现的，不是“普通卡牌对战 + 一个中间进度条”。  
而是：

**一个支持多牌同步提交、主议/旁议双区结算、中央议题持续推进、并每 3 轮触发一次引爆判定的叙事辩论式战斗系统。**
