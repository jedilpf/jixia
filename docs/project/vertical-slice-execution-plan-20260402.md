# 核心闭环推进方案（7天执行版）

日期：2026-04-02  
负责人：你（产品/总策）+ 我（执行与质量闸门）

## 1. 目标与边界

### 1.1 本阶段唯一目标
在 7 天内做出并验证一个稳定、可重复体验的 20-30 分钟闭环：

`章节开场 -> 3 场辩论 -> 选奖励卡 -> 章节结算`

### 1.2 本阶段硬约束
- 卡牌总量保持 170，不允许回退到 48。
- 不新增大系统（养成、联网、账号、后端）；
- 只修闭环相关问题：节奏、数值、可读性、关键交互；
- 每次改动后必须通过 `gate:daily`。

## 2. 验收指标（必须量化）

### 2.1 核心指标
- 新手首局完成率：`>= 70%`
- 单局时长（首局）：`20-30 分钟`
- AI 对战胜率区间：`45%-55%`
- 软锁/卡死/无法继续推进：`0 次`

### 2.2 体验指标
- 玩家“看不懂该做什么”的反馈比例：`<= 20%`
- 玩家“卡手（可用牌少/无意义）”反馈频次：每局 `<= 2 次`

## 3. 7天执行节奏

### Day 1（今天）：建立基线与样本
- 运行基线验证：`npm run typecheck`、`npm run lint`、`npm run gate:daily`
- 启动项目（固定命令）：`npx vite --host`
- 做 3 局人工试玩（同一版本），填写 `playtests/2026-04-02-session-01.md`
- 输出问题清单，按 P0/P1/P2 分级：
  - P0：卡死、崩溃、规则错误
  - P1：节奏明显拖沓、关键反馈缺失、数值畸形
  - P2：文案、视觉细节、非阻塞体验问题

### Day 2：闭环最小修正（只动 P0/P1）
- 修复 P0 全部问题；
- 修复最影响完成率的前 3 个 P1；
- 回归验证并记录修复前后差异。

### Day 3：卡牌可玩分层（170 张不变）
- 按“前期可用 / 中期核心 / 后期高风险”做使用层级标记；
- 检查前 5 回合抽牌是否可打、可选、可理解；
- 重点处理“前期废牌感”和“高费卡过早上手”。

### Day 4：AI 体验收敛
- 优先改善 AI 的“明显失误”而非追求复杂博弈；
- 目标是让玩家感知到 AI 有防守/反击意识；
- 做 10 局快测，记录胜率和异常局。

### Day 5：新手引导与反馈文案
- 补齐关键提示（阶段目标、可执行动作、收益预览）；
- 减少“玩家不知道下一步做什么”的节点；
- 做 5 局新手路径回放。

### Day 6：稳定性与边界测试
- 极端路径（保守出牌/激进出牌/连续失败）各测 2 局；
- 检查软锁、异常状态、资源越界；
- 整理“上线前必须修复清单（Blocking List）”。

### Day 7：验收与冻结
- 汇总本周数据，判断是否达到 2.1 指标；
- 达标则冻结闭环版本，进入“系统文档 + UI框图”阶段；
- 未达标则只补 Blocking List，不开新功能。

## 4. 每日固定工作流（你要求的“不可随意删减”）

1. 先跑健康检查：`typecheck -> lint -> gate:daily`  
2. 再做当日小目标（单一目标，不并行发散）  
3. 完成后做最少 2 局回归试玩并记录  
4. 记录“改了什么 -> 指标变化 -> 剩余风险”  
5. 当日收工前更新下一步待办（不清空历史）

## 5. 重要节点汇报机制

以下节点必须汇报后再继续下一步：
- 节点A：Day 1 基线问题清单完成
- 节点B：Day 2 P0 清零完成
- 节点C：Day 4 AI 10局结果出炉
- 节点D：Day 7 验收结论（达标/不达标）

## 6. 风险与止损

- 若胜率始终偏离 45%-55%：先冻结新改动，只做数值回调；
- 若单局时长 > 30 分钟：优先压缩回合冗余，不加新动画；
- 若新手完成率 < 70%：优先做流程提示，不先做高级系统；
- 若出现多AI并行改动冲突：先做冲突仲裁记录，再实施代码修改。
