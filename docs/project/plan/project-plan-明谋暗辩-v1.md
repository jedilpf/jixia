## 二、开发阶段规划

### 阶段一：核心机制完善（P0）

#### 任务1.1：完善信息隐藏机制

**目标**：确保提交阶段对方只能看到"放了牌到哪个槽位"，看不到具体卡牌

**修改文件**：
- `src/battleV2/types.ts` - 添加公开状态类型
- `src/battleV2/engine.ts` - 实现信息过滤逻辑
- `src/components/BattleFrameV2.tsx` - UI显示优化

**技术要点**：
```typescript
// 新增类型：对方可见的规划信息
interface PublicPlanInfo {
  hasMainCard: boolean;      // 是否有主论牌
  hasResponseCard: boolean;  // 是否有反制牌
  hasSecretCard: boolean;    // 是否有暗策牌
```

### 阶段二：结算流程规范化（P1）

#### 任务2.1：规范化结算顺序

**目标**：严格按照文档要求的顺序结算

**文档要求的结算顺序**：
1. 立论进场
2. 施策生效
3. 议位胜负结算
4. 根基变化/离场
5. 大势/筹更新
6. 胜利检查

**修改文件**：
- `src/battleV2/engine.ts` - 重构 resolveRound 函数

**技术要点**：
```typescript
function resolveRound(state: DebateBattleState): void {
  // Phase 1: 立论进场
  resolveThesisEntry(state);
  
  // Phase 2: 施策生效
  resolveStrategyEffects(state);
  
  // Phase 3: 议位胜负结算
  resolveSeatBattles(state);
  
  // Phase 4: 根基变化/离场
  resolveUnitDeath(state);
  
  // Phase 5: 大势/筹更新
  updateDaShiAndChou(state);
  
  // Phase 6: 胜利检查
  checkVictory(state);
}
```

#### 任务2.2：实现大势胜利条件

**目标**：先到8大势获胜（替代当前的心证归零）

**修改文件**：
- `src/battleV2/types.ts` - 添加大势字段
- `src/battleV2/engine.ts` - 实现大势计算和胜利判定

**技术要点**：

## 六、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 结算顺序冲突 | 高 | 编写详细单元测试，确保顺序正确 |
| 信息泄露 | 高 | 严格区分公开状态和私有状态 |
| AI决策不合理 | 中 | 优化AI决策算法，增加策略多样性 |
| UI动画卡顿 | 中 | 使用CSS动画，避免JS计算阻塞 |
| 网络同步问题 | 低 | 当前为单机版，暂不考虑 |

## 七、验收标准

### 7.1 功能验收

- [ ] 明辩阶段对方只能看到"有牌/无牌"和目标路位
- [ ] 暗策阶段对方只能看到"有牌/无牌"和目标路位
- [ ] 揭示阶段双方同时看到完整卡牌
- [ ] 结算顺序严格按照文档要求执行
- [ ] 大势达到8时判定胜利
- [ ] 筹机制正常工作

### 7.2 性能验收

- [ ] 单次结算耗时 < 100ms
- [ ] UI动画流畅，无卡顿
- [ ] 内存占用稳定，无泄漏

### 7.3 体验验收

- [ ] 操作流程清晰易懂
- [ ] 状态变化有明确提示
- [ ] 胜负判定准确无误
