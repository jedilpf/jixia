# battleV2 三阶卡牌战斗运作模型（2026-04-08）

## 1. 目的
- 让 `1阶 / 2阶 / 3阶` 不只是“解锁等级不同”，而是在战斗内有清晰的职责分工。
- 保持设计完全挂接在当前 `battleV2` 骨架上，而不是引入外部升级循环。
- 给后续数值、技能重写、运行时实现提供统一方向。

## 2. 当前结构锚点
- 回合骨架固定为：`ming_bian -> an_mou -> reveal -> resolve`
- 当前系统里已经存在可复用锚点：
  - `topic`
  - `writing / wenMai`
  - 三路席位与路权奖励
  - `playerLevel`
  - `deck tier quota`
- 当前五类卡牌分别对应：
  - `立论`
  - `策术`
  - `玄章`
  - `门客`
  - `反诘`

## 3. 当前卡池观察
- 1阶样例更像“底盘牌”：
  - `温言立论`
  - `竹牍抄录`
  - `讲席清规`
  - `断简成法`
- 2阶样例已经开始出现“节奏转换”和“条件收益”：
  - `司史执笔`
  - `兼济匠师`
  - `太清符箓`
  - `两仪折冲`
- 3阶样例明显更接近“改局牌”：
  - `一纸敕令`
  - `破阵天雷`
  - `千机壁垒`
  - `锋将·破军`
- 这说明当前卡牌文本本身已经天然支持“三阶职责化”，不需要额外引入外部式升级结构。

## 4. 一个必须先统一的实现前提
- 当前 `showcaseCards` 的解锁层级口径，与 `battleV2` 运行时的星级推导口径并不完全一致。
- 参考位置：
  - [showcaseCards.ts](/f:/zz/jixia2.0【完整链路版本】/src/data/showcaseCards.ts#L1)
  - [tierSystem.ts](/f:/zz/jixia2.0【完整链路版本】/src/battleV2/tierSystem.ts#L41)
  - [cards.ts](/f:/zz/jixia2.0【完整链路版本】/src/battleV2/cards.ts#L270)
- 结论：
  - 在真正把三阶职责做进运行时前，必须先统一“展示层三阶”和“战斗层三阶”的口径。
  - 否则玩家看到的阶级身份，会和战斗实际判定错位。

## 5. 三阶战斗职责总模型

| 阶级 | 战斗身份 | 主要时机 | 主要价值 | 设计硬约束 |
|---|---|---|---|---|
| 1阶 | 底盘牌 | `ming_bian` 前半段、基础结算 | 稳定展开、补手、保底防守、基础站位 | 文本短、条件少、收益稳定，不做大幅 swing |
| 2阶 | 节奏牌 | `ming_bian` 后半段、`an_mou` 过渡、下回合承接 | 转速、连动、拆招、转换路权 | 至多 1 个中等条件，不应一张牌同时完成过多任务 |
| 3阶 | 改局牌 | `reveal` 前后、`resolve` 关键窗、特定 topic / writing 条件下 | 局势翻转、压轴斩获、关键路锁定 | 数量少、窗口明确、前置条件清晰，不能做无条件全能牌 |

## 6. 三阶分别该怎么“在 battleV2 里运作”

### 6.1 1阶卡：负责把局面跑起来
- 1阶卡的核心不是弱，而是“可预测、可重复、可铺底”。
- 它们优先服务以下目标：
  - 建立第一个可行动回合
  - 保证手牌不断档
  - 在三路里至少拿到一个可争夺点
  - 给 2 阶和 3 阶创造条件
- 1阶卡最适合挂接的系统变量：
  - 已使用同门牌
  - 是否受伤
  - 本回合第一次使用某类牌
  - 当前是否已存在玄章/门客
- 1阶卡不应该承担：
  - 一张牌直接清场
  - 一张牌同时拿手牌、护持、路权、爆发伤害
  - 长回合的滚雪球核心

### 6.2 2阶卡：负责把局面转起来
- 2阶卡是本系统真正的“中盘发动机”。
- 它们应该承担三类工作：
  - 把 1 阶的稳定动作转成更高收益
  - 在 `ming_bian -> an_mou` 之间制造信息差
  - 为 3 阶卡搭桥，但自己不能直接等同于终结技
- 2阶卡最适合挂接的系统变量：
  - 当前议题偏好
  - 上回合/本回合是否着书
  - 指定席位是否已有友军
  - 敌方是否带负面
  - 是否已控制某一路
- 2阶卡的理想手感：
  - 打出后，玩家能明显感觉“节奏被我抓到了”
  - 但还没到“这把已经结束”的程度

### 6.3 3阶卡：负责改写关键回合
- 3阶卡不是“更大数字”，而是“更强时机价值”。
- 它们应当只在以下时机显著强：
  - 你已经围绕某一路做了准备
  - 当前 `topic` 明显偏向某类效果
  - 你已经完成 `writing` 铺垫
  - 双方场面已经有负面状态、玄章层、门客密度等可利用条件
- 3阶卡不应该默认提供：
  - 无条件封锁全回合
  - 无准备的一键逆天翻盘
  - 与所有 topic / 路线都同样高效的通吃收益
- 3阶卡的理想体验是：
  - 读局正确时，它像“压轴论证”
  - 读局错误时，它也应该付出节奏代价

## 7. 五类卡牌在三阶中的推荐职责

### 7.1 立论
- 1阶立论：
  - 负责开场压题、补底蕴、轻微补手
  - 条件最好只依赖“本回合第一张 / 同门 / 未受伤”这类简单信息
- 2阶立论：
  - 负责把已有场面转成优势
  - 可以开始吃 `topic`、路权、门客在场等中等条件
- 3阶立论：
  - 负责“定论式 swing”
  - 重点不是纯直伤，而是对某一路、某轮 `reveal/resolve` 的决定性影响

### 7.2 策术
- 1阶策术：
  - 负责检索、修手、基础效率交换
- 2阶策术：
  - 负责节奏切换、净化、点控、转费、转资源
- 3阶策术：
  - 负责压轴处理
  - 例如清玄章、区域爆发、对某类资源结构的重击
  - 但必须绑定条件，不做所有场面都最优

### 7.3 玄章
- 1阶玄章：
  - 负责底盘护持、被动积累、轻微路线偏置
- 2阶玄章：
  - 负责建立“持续收益引擎”
  - 例如某类牌首用收益、两回合窗口、门客保护
- 3阶玄章：
  - 负责定义局面规则
  - 但只能偏科，不能同时兼顾防守、抽牌、封锁、斩杀
  - 3阶玄章要更像“场上议程”而不是“常驻万能增益”

### 7.4 门客
- 1阶门客：
  - 负责补位、保路、做基础站场
- 2阶门客：
  - 负责带一个明确入场功能
  - 例如检索、施压、沉默、保护
- 3阶门客：
  - 负责“强存在感单位”
  - 强度应更多来自击败、驻场威胁、席位联动
  - 而不是裸面板无限膨胀

### 7.5 反诘
- 1阶反诘：
  - 负责给基础对抗感
  - 低费、短文本、只拦一次关键动作
- 2阶反诘：
  - 负责打信息差
  - 可以附带弃牌、沉默、延迟损益
- 3阶反诘：
  - 不宜大量存在
  - 若存在，必须是“高门槛、强识别”的判定牌
  - 避免把 3 阶系统做成纯 counter 压制环境

## 8. 三阶与 phase 的推荐绑定方式

| 时机窗 | 推荐主阶级 | 设计含义 |
|---|---|---|
| `ming_bian` 起手窗 | 1阶 | 建立节奏、铺底盘、给 topic 初始响应 |
| `ming_bian` 连段窗 | 1阶 / 2阶 | 把基础动作串起来，但不直接终局 |
| `an_mou` 埋伏窗 | 2阶 | 制造信息差、准备反手、改变下层收益 |
| `reveal` 判定窗 | 2阶 / 3阶 | 兑现隐藏条件、把准备转成 swing |
| `resolve` 收束窗 | 3阶 | 把路权、topic、writing、负面状态等条件一次性兑现 |

结论：
- 1阶主要解决“能不能跑起来”。
- 2阶主要解决“怎么把优势做出来”。
- 3阶主要解决“什么时候把优势结算掉”。

## 9. 三阶与 topic / writing / 三路 的挂接建议

### 9.1 与 topic 的关系
- 1阶卡：
  - topic 只做轻度加权，不改它们的底层职责。
- 2阶卡：
  - 可以开始出现“在某议题下更顺”的文本方向。
- 3阶卡：
  - 应明显吃议题窗口，但不能所有 3 阶都只吃同一种议题。

### 9.2 与 writing 的关系
- `writing` 已经能带来 `wenMai` 与 `maxLingShi` 增长。
- 因此推荐：
  - 1阶卡只把着书当成轻条件
  - 2阶卡把着书当“承接器”
  - 3阶卡把着书当“压轴许可”
- 也就是说，`writing` 更适合作为三阶卡的前置许可，而不是独立升级系统。

### 9.3 与三路的关系
- 1阶卡：
  - 负责占位、保位、基础抢路
- 2阶卡：
  - 负责把一路优势扩成两路节奏
  - 或把一路劣势止损
- 3阶卡：
  - 负责把某一路准备好的条件兑现为关键回合结果
  - 不建议做“无条件三路全扫”的常规化设计

## 10. 面向实现的最小附加信息建议
- 当前运行时只有 `effectKind/effectValue/power/hp`，不足以稳定表达“三阶职责窗口”。
- 如果未来要落实现，最小可增量字段建议如下：

| 字段 | 说明 | 例子 |
|---|---|---|
| `tierRole` | 该卡在本阶的战斗职责 | `stabilizer / pivot / swing` |
| `timingWindow` | 主要生效窗口 | `ming_bian_open / combo / an_mou / reveal / resolve` |
| `conditionTags` | 触发前提标签 | `topic_match`, `writing_ready`, `enemy_debuffed`, `lane_ahead` |
| `payoffShape` | 收益形状 | `immediate`, `delayed`, `conditional_swing` |
| `telegraphLevel` | 对手是否应可预判 | `low`, `medium`, `high` |

说明：
- 这些字段是“够用的元数据”，不是新资源系统。
- 它们不会引入外部结构，只是帮助我们把现有 battleV2 的牌写得更清楚。

## 11. 后续实现顺序建议
1. 先统一展示层与战斗层的阶级口径。
2. 再给少量代表卡补 `tierRole / timingWindow / conditionTags` 草案。
3. 先选每阶 5-8 张卡做小样本改写，不要一次性改完整池。
4. 用模拟验证三个问题：
   - 1阶是否足够稳定开局
   - 2阶是否真的在中盘形成节奏差
   - 3阶是否只在关键窗口强，而不是全程都强
5. 验证通过后，再批量扩写到完整卡池。

## 12. 一句话结论
- 三阶系统接下来不该继续围绕“怎么升级”展开，而该围绕“1阶铺底、2阶转速、3阶改局”展开。
- 这套模型已经可以完全挂在我们现有 `battleV2` 的 phase、topic、writing、三路和配额系统上，不需要引入任何外部同构结构。
