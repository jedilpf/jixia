# 系统策划背书文档（前后端边界 / XMind系统文档 / Axure框图 / 竞品拆解）

- 文档版本：v1.0
- 产出日期：2026-04-01
- 适用项目：稷下类“卡牌对战 + 图鉴收集 + 剧情推进”产品
- 使用对象：策划、客户端、服务端、UI/UX、测试、制作人

## 1. 文档定位

本文件用于回答三个核心问题：

1. 系统策划工作如何分层，尤其是前后端边界如何划清。
2. 如何把一个“系统想法”落成标准产物：XMind结构文档 + Axure框图 + UI需求文档。
3. 竞品为什么能火，我们能借什么、不能照抄什么、如何改造成自己的版本。

目标不是“写一份好看的方案”，而是形成能直接指导开发排期与验收的生产文档。

## 2. 前后端职责划分（先定边界，再做系统）

## 2.1 总原则

- 前端负责“表现、交互、即时反馈”。
- 后端负责“权威状态、可追溯一致性、安全和可扩展”。
- 策划文档必须显式标记每条规则的“执行责任方”。

## 2.2 责任矩阵

| 模块 | 前端主责 | 后端主责 | 共同约束 |
|---|---|---|---|
| 卡牌图鉴展示 | 页面布局、筛选排序交互、动画反馈、红点展示 | 图鉴状态权威存储、领取/解锁判定、幂等接口 | 字段协议统一（ID、状态、更新时间） |
| 对战流程 | 回合内UI、操作引导、特效与日志展示 | 对战结算权威、反作弊校验、战报落库 | 战斗状态机字段一致 |
| 剧情系统 | 章节入口、节点分支展示、阅读体验 | 存档点、分支条件校验、进度同步 | 存档协议版本兼容 |
| 红点系统 | 前端聚合显示与清除动画 | 红点源头事件计算与订阅下发 | 红点优先级和清除条件一致 |
| 经济与奖励 | 展示与引导、领取按钮态 | 发奖校验、防重发、流水记录 | 奖励ID与发放日志可追溯 |

## 2.3 系统文档的“责任标注”规范

每条规则后加标签：

- `[FE]` 前端执行
- `[BE]` 后端执行
- `[FE+BE]` 双端协同
- `[策划验收]` 策划必须验收的行为结果

示例：

- “图鉴新卡首次解锁后，入口显示一级红点。”`[BE]`
- “当玩家进入图鉴主页后，入口红点延迟 300ms 消失并播放消隐动画。”`[FE]`
- “红点消失必须在后端状态已确认‘已读’后触发，避免刷新后反复出现。”`[FE+BE]`

## 3. 选定系统样例：人物志-卡牌图鉴-对战联动系统

本节用一个完整系统举例，作为你团队后续仿写模板。

## 3.1 设计目的

- 目的1：把“收集感”从静态卡牌列表提升为“人物志 + 卡牌语义 + 战斗实用性”三位一体。
- 目的2：让图鉴不仅是展示页面，而是战斗准备链路的一部分（看卡 -> 配卡 -> 开战）。
- 目的3：为中长期内容扩展留接口（新门派、新人物、新章节、新活动词条）。

## 3.2 功能概述

- 人物志页：按门派/角色展示人物卡档案与关联卡牌。
- 卡牌图鉴页：按门派、稀有度、类型查看全部卡牌；支持详情翻页。
- 联动入口：图鉴详情支持“加入预组”“前往对战”。
- 解锁来源：剧情推进、任务奖励、活动兑换、商店获得。
- 红点链路：新解锁人物/卡牌、可领取图鉴奖励、未读剧情片段提示。

## 3.3 系统规则

### 3.3.1 数据规则

- 卡牌主键唯一：`card_id`。
- 人物主键唯一：`character_id`。
- 人物与卡牌关系：`1:N`（一个人物可关联多张卡）。
- 图鉴状态：`locked / unlocked / mastered`。
- 当前项目基线：图鉴与战斗演算统一使用全量卡池（当前 170 张）`[策划验收]`。

### 3.3.2 解锁规则

- 初始状态：玩家进入系统时仅解锁新手引导卡组与基础人物志。
- 解锁条件：
  - 完成指定剧情节点。
  - 达成指定对战胜场。
  - 领取特定活动奖励。
- 重复获得规则：重复卡转化为养成素材（或碎片）并记录来源。

### 3.3.3 展示规则

- 已解锁卡牌展示完整信息。
- 未解锁卡牌展示剪影、基础门派标记、解锁来源提示。
- 人物志页面默认按“剧情推荐顺序”，支持切换“战力/稀有度/更新时间”。

### 3.3.4 联动规则

- 从图鉴进入对战时，默认带入最近一次使用的门派配置。
- 图鉴内“加入预组”操作成功后，必须即时反馈并可撤销。
- 对战结束后，若触发新解锁，返回大厅时必须能看到入口红点。

## 5. 红点规则（重点）

红点做不好会直接伤害体验，必须单独成章。

## 5.1 红点层级

- 一级红点：主菜单入口（人物志/图鉴）显示。
- 二级红点：子页签（人物、卡牌、奖励）显示。
- 三级红点：具体条目（某人物、某卡牌）显示。

## 5.2 触发源

- `new_unlock_character`：新人物解锁。
- `new_unlock_card`：新卡牌解锁。
- `codex_reward_claimable`：图鉴奖励可领取。
- `story_note_unread`：关联剧情未读。

## 5.3 清除规则

- 一级红点不因“打开入口”直接清除，需对应子红点全部清除后消失。
- 条目红点在条目详情停留达到阈值（建议 >= 800ms）后清除。
- 可领取奖励红点仅在领取成功后清除，不因点击弹窗清除。

## 5.4 容错规则

- 本地缓存与服务器状态冲突时，以服务器为准并触发静默重拉。
- 红点清除请求失败时，前端不直接清除，避免“假清除”。

## 6. 界面规则（用于 UI 和前端共识）

## 6.1 信息层级

- 第一层：入口价值信息（总收集进度、可领奖数、最新解锁）。
- 第二层：筛选信息（门派、类型、稀有度、状态）。
- 第三层：条目详情（技能、背景、关联人物、实战建议）。

## 6.2 交互规则

- 卡片点击：进入详情页，保留上次滚动位置。
- 左右翻页：翻页动效应可中断，避免连击卡死。
- 筛选切换：500ms 内完成响应，超时必须有 loading 提示。

## 6.3 视觉反馈规则

- 可交互元素最小点击区域 >= 36px。
- 红点动画时长 180ms~260ms，避免高频闪烁。
- 未解锁卡牌统一使用同一套遮罩与文案模板。

## 7. Axure 框图产出规范（你可以直接照这个做）

## 7.1 页面树建议

1. `00_信息架构总览`
2. `01_人物志主页`
3. `02_图鉴主页`
4. `03_卡牌详情`
5. `04_图鉴奖励弹窗`
6. `05_加入预组弹窗`
7. `06_异常态（空态/断网/加载失败）`

## 7.2 每页必须包含

- 页面目标一句话。
- 核心模块分区（顶部区/筛选区/内容区/底部操作区）。
- 关键组件命名（统一前缀，如 `Codex_`）。
- 交互说明（点击、滑动、返回、弹窗关闭）。
- 状态说明（默认、空态、加载、失败、已完成）。

## 7.3 Axure 标注规范

- 所有按钮标注“触发条件 + 成功结果 + 失败结果”。
- 所有列表标注“分页/懒加载策略”。
- 所有动态内容标注“字段来源（FE缓存/BE接口）”。
- 所有红点标注“触发源和清除条件”。

## 8.2 元件清单

| 元件ID | 类型 | 文案/资源 | 状态 | 交互 | 数据来源 |
|---|---|---|---|---|---|
| `Codex_Filter_Faction` | 筛选器 | 门派标签 | 默认/选中/禁用 | 点击切换 | FE本地 + BE配置 |
| `Codex_Card_Item` | 卡片 | 卡牌缩略图+名称 | 锁定/解锁/精通 | 点击进详情 | BE权威 |
| `Codex_RedPoint_Entry` | 红点 | 数字/点 | 显示/隐藏 | 自动更新 | BE事件流 |

## 8.3 交互流程

1. 用户进入图鉴主页。
2. 系统拉取图鉴状态并渲染首屏。
3. 用户切换筛选条件。
4. 用户点击卡牌进入详情。
5. 用户执行“加入预组”。
6. 系统返回结果并刷新状态。

## 8.4 异常处理

- 接口超时：显示“重试”按钮与兜底文案。
- 数据为空：显示空态插画与引导按钮。
- 版本不匹配：显示升级提醒并阻断关键操作。

## 8.5 验收条目

- 功能验收：主路径完整可走通。
- 交互验收：关键反馈时序正确。
- 数据验收：字段显示与后端一致。
- 体验验收：红点不误触发、不误清除。

## 9. 竞品拆解：从“火”到“可迁移能力”

## 9.1 拆解框架（建议固定）

1. 核心循环：玩家 30 秒内在做什么决策。
2. 短期目标：玩家 10 分钟内能获得什么正反馈。
3. 中期目标：玩家 7 天内是否形成“可持续追求”。
4. 货币化点：付费是否解决效率或表达，不破坏公平。
5. 社交/传播点：内容是否可分享、可比较、可讨论。
6. 内容生产效率：新内容上线成本是否可控。

## 9.2 竞品拆解样例（方法型，不依赖单一数值）

| 竞品 | 关键成功点 | 可借鉴能力 | 我们如何改造后再用 |
|---|---|---|---|
| 炉石传说（Hearthstone） | 卡牌关键词清晰、回合反馈强 | 文案标准化 + 关键字系统 | 建立“术语词典+效果标签”，减少学习成本 |
| 漫威终极逆转（Marvel Snap） | 短局、三路博弈、翻盘时刻密集 | 快节奏回合与高信息密度 | 保留三路思路，但结合“明辩/暗谋”形成本项目叙事特色 |
| 杀戮尖塔（Slay the Spire） | 构筑成长和风险选择持续驱动 | 中期成长路径与构筑差异 | 用“人物志羁绊 + 门派构筑”替代纯数值爬塔 |
| 阴阳师百闻牌 | 角色IP与卡牌关系强 | 人物叙事绑定卡牌价值 | 做“人物志 -> 关联卡 -> 剧情碎片”闭环 |

## 9.3 “拿来用”判断标准

一个机制只有同时满足以下条件才建议引入：

- 与本项目核心循环不冲突。
- 能在 2~3 个版本内被团队稳定生产。
- 不引入不可控的数值债或内容债。
- 能被当前用户群体快速理解。

## 9.4 经验问题如何制度化

把“经验”变成“模板资产”：

- 建立《竞品机制迁移卡》模板（机制名、价值、风险、改造方案、验收指标）。
- 每次只迁移 1~2 个关键机制，避免一次性大改。
- 迁移前先做小流量验证，验证不过不进主线版本。

## 10. 作为内容背书的结论（可给团队/管理层）

1. 我们不是在“抄系统”，而是在迁移“可验证的成功机制能力”。
2. 系统策划的核心产物不是脑图本身，而是“可执行规则 + 可验收状态 + 可扩展接口”。
3. 前后端边界先划清，能大幅降低返工成本和跨团队误解成本。
4. 红点规则、异常规则、联动规则如果不先定，后续体验问题会指数级放大。
5. Axure 与 UI-PRD 不是形式，而是把策略意图落地为研发可实现的最短路径。

## 11. 交付清单（建议按此验收策划产物）

- `系统文档`：包含设计目的、功能概述、系统规则、红点规则、界面规则。

- `Axure框图`：完整页面树 + 状态页 + 关键交互标注。
- `UI需求文档`：元件清单、字段来源、异常态、验收条目。
- `竞品拆解报告`：至少 3 款竞品，且每款都有“可借鉴 + 改造方案 + 风险”。

## 12. 附：快速评审问题（评审会上直接用）

1. 这个系统的核心目标是否一句话讲得清？
2. 前后端对“权威状态”的归属是否一致？
3. 红点触发与清除是否有冲突场景？
4. 若接口失败，页面会怎样，玩家会看到什么？
5. 竞品机制迁移是否经过“改造”而非“照搬”？
6. 有没有定义上线后 1 周内的观测指标？
7. 本系统是否真正服务于主循环，而不是新增负担？

这 7 个问题若都能答清，本系统可以进入开发排期。
