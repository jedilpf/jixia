## 三、一比一游戏方案的联网可行性评估

### 3.1 核心技术挑战

#### Challenge 1: 严格的时间同步
**等级**：⚠️ 高

**问题**：
- 两个 25 秒的阶段必须在所有玩家之间**精确同步**
- 任何 > 5 秒的时间偏差都会完全破坏游戏体验
- 网络抖动、高延迟会导致"双方看到不同的倒计时"

**解决方案**：
```typescript
// 服务器生成中央时钟，所有客户端同步到该时钟
// 使用 NTP-like 协议校准本地时间
const serverTime = Date.now();
const clientOffset = localTime - serverTime;
// 后续所有倒计时基于服务器时间计算
```

**参考**：
- 游戏服务器尽量靠近用户（CDN 节点）
- 使用自适应时间补偿算法
- 接受 ±1s 的同步误差，超过则暂停比赛

---

#### Challenge 2: 同步提交的一致性
**等级**：⚠️ 高

**问题**：
- 必须保证 A 玩家提交的卡牌内容，B 玩家能100%接收
- 网络包丢失 → 重传 → 时间不够 → 超时处理

**解决方案**：
```typescript
// 1. 玩家在明辩阶段提交卡牌
submitAction(action) {
  // 发往服务器，带时间戳
  socket.emit('action', { action, timestamp, clientId });
}

// 2. 服务器收集双方提交
// 3. 揭示阶段开始前，服务器确认双方都已提交
// 4. 若超时，按"无操作"或"上一步操作"处理
```

**参考**：
- 设置 2s 超时做最后冲刺缓冲
- 明确定义"超时操作"的降级行为
- 明牌阶段允许"改口"，但暗谋阶段不允许

---

#### Challenge 3: 防作弊与反向工程
**等级**：⚠️ 中等

**问题**：
- 玩家可以本地修改 JS 代码，看到对方暗策
- 本地时间可以被修改，影响公平性

**解决方案**：
```typescript
// 1. 所有关键逻辑在服务器上执行
// 2. 客户端只负责显示与输入
// 3. 服务器验证每一步操作的合法性

validateAction(player, action) {
  // 检查费用、手牌、目标合法性等
  if (!isLegal(action)) {
    reject();
  }
}

// 4. 暗策内容完全由服务器管理，不下发给客户端
```

---

#### Challenge 4: 跨地域延迟
**等级**：⚠️ 中等

**观察**：
- 国内玩家匹配国外玩家 → RTT 可能 300ms+
- 此时双阶段制的优势减弱（等待时间拉长）

**解决方案**：
1. **地域分服**：优先匹配同地区玩家
2. **延迟容限**：允许一定范围的延迟变化
3. **补偿机制**：高延迟玩家获得微小的"思考时间加成"

### 3.3 风险与建议

#### 🔴 高风险

| 风险 | 影响 | 缓解方案 |
|------|------|----------|
| 网络延迟 | 游戏核心体验崩溃 | 服务器就近部署、地域分服 |
