## 错题本

### ❌ 错误 #1：误以为 Edit 工具修改已落盘

**时间**：2026-04-09
**场景**：修复社区模块代码后，用户询问是否有未提交改动
**问题**：通过 `SearchReplace` 工具修改了代码，但 `git status` 显示没有未提交修改
**原因**：误以为 Edit 工具的修改会自动保存到磁盘
**教训**：
- 修改后必须通过 `Read` 或 `Grep` 验证修改是否生效
- 不能仅凭工具调用成功就假设修改已落盘
- 重要修改后要主动验证文件状态

---

### ❌ 错误 #2：修复脚本引入新的语法错误

**时间**：2026-04-09
**场景**：修复 `prolog.ts` 编码问题时
**问题**：修复脚本将单引号字符串错误地改成了双引号结尾
**原因**：
- 脚本逻辑过于简单，没有考虑引号匹配
- 批量替换时没有区分单双引号
**教训**：
- 批量修复脚本必须经过测试验证
- 对于语法敏感的修复，应该逐行检查
- 修复后必须运行 TypeScript 检查验证

---

## 项目规范

### AGENTS 规则遵守

1. **Task 流程**：创建 task JSON → validate → prompt → 产出 → finalize
2. **UNKNOWN 标注**：无法确认的信息必须标注为 UNKNOWN
3. **不虚构数据**：不得虚构团队实名、融资条款、DAU/留存等数据
4. **冲突仲裁**：遇到冲突先创建仲裁记录，不擅自决定

### 文件修改规范

1. **修改前**：
   - 确认文件是否在受保护列表
   - 阅读相关代码了解上下文
   - 确认修改范围

2. **修改中**：
   - 保持原有代码风格
   - 不删除无关代码
   - 添加必要的注释

3. **修改后**：
   - 验证修改生效
   - 运行 TypeScript 检查
   - 测试功能正常

## 给其他AI的建议

### 代码修复

1. **验证修改生效**：修改后务必通过 `Read` 或 `GetDiagnostics` 验证
2. **TypeScript 检查**：修改后运行 `npx tsc --noEmit` 确保无类型错误
3. **测试运行**：重要修改后启动开发服务器验证

### 文件编码

1. **中文文件注意编码**：Windows 环境容易出现 UTF-8 编码问题
2. **发现 `�` 字符**：说明文件编码已损坏，需要全面修复
3. **修复策略**：使用 Node.js 脚本处理，避免 PowerShell 编码问题

### 与用户的沟通

1. **明确约束**：重要修改前确认用户约束（如争鸣史保护规则）
2. **及时汇报**：完成关键任务后主动汇报结果
3. **承认错误**：发现问题及时承认并修复，不隐瞒
4. **UI修改**：先确认风格方向，提供选项让用户选择
